## 1. 문서 목적

본 문서는 `00-project-overview.md`, `01-screen-structure.md`, `02-db-design.md`, `03-api-spec.md`, `04-scale-json.md`, `05-backend-architecture.md`, `06-frontend-architecture.md`, `07-validation-rules.md`, `08-error-handling.md`를 기준으로,

본 문서의 목적은 다음과 같다.

## 2. 테스트 설계 원칙

### 2.1 핵심 업무 흐름 우선 원칙
이 시스템의 최우선 테스트 대상은 아래 흐름이다.

1. 로그인
2. 대상자 검색/등록
3. 대상자 상세 진입
4. 검사 시작
5. 척도 선택
6. 척도 입력
7. 세션 저장
8. 세션 상세 확인
9. 오입력 처리
10. 통계/기록 조회

즉, 예쁘게 보이는 UI보다 **저장 정확성, 권한 정확성, 통계 정확성**을 우선 검증한다.

### 2.2 세션 저장 최우선 원칙
가장 중요한 테스트 대상은 `검사 세션 저장` 이다.

이유:
- 여러 척도를 한 번에 저장한다.
- 서버가 최종 점수/판정/경고를 계산한다.
- 세션/척도결과/문항응답/경고를 하나의 트랜잭션으로 저장한다.
- 저장 실패 시 전체 롤백되어야 한다.

### 2.3 상태값 기반 회귀 테스트 원칙
이 시스템은 삭제 대신 상태값(`MISREGISTERED`, `MISENTERED`)으로 처리하므로,  
기능 테스트는 반드시 “정상 상태”뿐 아니라 “상태 변경 이후”까지 포함해야 한다.

### 2.4 권한 테스트 분리 원칙
같은 기능도 아래 사용자 유형별로 분리 검증한다.

- 비로그인 사용자
- 일반 사용자
- 작성자
- 관리자
- 승인 대기 사용자
- 비활성 사용자

### 2.5 척도 계산 독립 검증 원칙
척도 계산은 UI 테스트와 별개로,  
반드시 **입력 → 점수 → 판정 → 경고**를 독립 검증해야 한다.

### 2.6 기본 제외 정책 검증 원칙
아래 데이터는 기본 화면에서 숨겨져야 한다.

- `MISREGISTERED` 대상자
- `MISENTERED` 세션/기록

따라서 “생성 성공” 테스트와 함께 “기본 목록에서 숨김” 테스트를 반드시 짝으로 확인한다.

## 5. 우선순위 정의

- **P0**
  - 배포 전에 반드시 통과해야 하는 핵심 시나리오
  - 저장/권한/점수/상태 관련 치명 기능
- **P1**
  - 주요 화면과 운영 기능
  - 배포 전 강하게 권장되는 시나리오
- **P2**
  - 후속 회귀 테스트 또는 고도화 단계에서 강화할 시나리오

## 6. 테스트 환경 기준

## 6.1 기본 환경
- 내부 개발 환경 또는 사내 테스트 서버
- 백엔드 + DB + 척도 JSON 로더가 모두 구동된 상태
- 테스트용 관리자 계정 1개 이상
- 테스트용 일반 사용자 계정 2개 이상

## 6.2 필수 테스트 계정
1. `adminA`
   - 관리자
   - ACTIVE

2. `userA`
   - 일반 사용자
   - ACTIVE

3. `userB`
   - 일반 사용자
   - ACTIVE

4. `pendingUser`
   - 일반 사용자
   - PENDING

5. `inactiveUser`
   - 일반 사용자
   - INACTIVE

6. `rejectedUser`
   - 일반 사용자
   - REJECTED

## 6.3 테스트 대상자 데이터 예시
1. 정상 대상자 A
   - ACTIVE
   - userA 작성

2. 정상 대상자 B
   - ACTIVE
   - userB 작성

3. 오등록 대상자 C
   - MISREGISTERED
   - userA 작성

4. 비활성 대상자 D
   - INACTIVE

## 6.4 테스트 세션 데이터 예시
1. 정상 세션 S1
   - COMPLETED
   - userA 작성

2. 정상 세션 S2
   - COMPLETED
   - alert 포함

3. 오입력 세션 S3
   - MISENTERED
   - userA 작성

## 10. 인증/세션 테스트 시나리오

## AUTH-P0-01 로그인 성공
- 우선순위: P0
- 목적: ACTIVE 계정의 정상 로그인 검증
- 사전조건: `userA` 계정 ACTIVE
- 실행절차:
  1. 로그인 화면 진입
  2. 올바른 아이디/비밀번호 입력
  3. 로그인 버튼 클릭
- 기대결과:
  - 로그인 성공
  - `/clients` 로 이동
  - 현재 사용자 정보가 상단에 표시됨
  - 세션 쿠키 생성
- 자동화 권장 여부: 예

## AUTH-P0-02 승인 대기 계정 로그인 차단
- 우선순위: P0
- 목적: PENDING 계정 로그인 차단 검증
- 사전조건: `pendingUser` 존재
- 실행절차:
  1. 로그인 시도
- 기대결과:
  - 로그인 실패
  - 승인 대기 안내 메시지 표시
  - 세션 생성 안 됨
- 자동화 권장 여부: 예

## AUTH-P0-03 비활성 계정 로그인 차단
- 우선순위: P0
- 기대결과:
  - 로그인 불가
  - 비활성 상태 안내 메시지 표시
- 자동화 권장 여부: 예

## AUTH-P0-04 반려 계정 로그인 차단
- 우선순위: P0
- 기대결과:
  - 로그인 불가
  - 반려 상태 안내 메시지 표시
- 자동화 권장 여부: 예

## AUTH-P0-05 잘못된 비밀번호 로그인 실패
- 우선순위: P0
- 기대결과:

- 기대결과:
  - 세션 만료
  - 로그인 화면 이동 또는 비로그인 상태 전환
- 자동화 권장 여부: 예

## 28. 바이브 코딩 시 테스트 구현 규칙

1. 세션 저장 기능부터 테스트를 먼저 쓴다.
2. 척도별 계산 테스트는 UI보다 먼저 통과시킨다.
3. 권한 테스트를 버튼 노출 테스트로 끝내지 않는다.
4. 상태값 변경 후 목록/통계 반영까지 한 번에 검증한다.
5. 프론트 preview 계산과 서버 저장 결과의 차이를 허용하되, 최종 서버 결과를 기준으로 테스트한다.
6. `MISREGISTERED`, `MISENTERED`, `PENDING`, `INACTIVE` 상태 시나리오는 항상 별도 테스트를 둔다.
7. 자동화 테스트 이름만 보고도 업무 흐름이 읽히게 작성한다.
8. 테스트 데이터는 서로 오염되지 않게 초기화한다.

## 29. 초안 기준 최종 권장 사항

- 이 프로젝트의 테스트 중심축은 `세션 저장`, `척도 계산`, `권한`, `상태값 제외 정책`이다.
- P0는 저장/권한/점수/상태/통계 정확성을 다루고, 반드시 자동화하는 것을 권장한다.
- 척도 계산은 화면 테스트와 분리된 독립 테스트 세트로 관리한다.
- 오입력/오등록은 생성 성공 테스트와 숨김 테스트를 반드시 짝으로 만든다.
- 통계 테스트는 오입력 제외 정책까지 확인해야 의미가 있다.
- 관리자 기능은 승인/로그/백업까지 최소 1회 실동작을 검증해야 한다.
- 배포 전에는 P0 100% 통과를 최소 기준으로 삼는다.
