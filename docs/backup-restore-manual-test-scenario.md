# 백업/복원 수동 검증 시나리오

## 0. 테스트 전 정리

- [ ] `RESTORE-A-*`, `RESTORE-B-*` 식별자가 이미 남아 있는지 먼저 검색한다.
- [ ] 기존 데이터가 남아 있으면 이번 회차 식별자에 짧은 suffix 또는 테스트 일시 축약값을 붙인다.
- 예시: `RESTORE-A-CLIENT-001-0405A`, `restore_a_user_0405a`
- [ ] suffix를 붙였다면 대상자 이름, 사용자 아이디, 연락처, 메모, 팀명 등 이번 문서의 A/B 예시값 전체에 같은 규칙을 일관되게 적용한다.
- [ ] 테스트 시작 전 기준 상태를 보여주는 스크린샷 1장을 확보한다.
- 권장 기준 스크린샷: 대상자 목록 또는 백업 관리 화면에서 현재 테스트 식별자가 없음을 보여주는 화면

## 1. 테스트 목적

이 시나리오는 **A 시점 백업으로 복원했을 때 B 시점 이후 변경사항이 제거되고 A 시점 상태가 복구되는지**를 검증한다.

이번 검증은 아래 2가지를 함께 확인해야 통과로 본다.

- B 시점 이후에 새로 생성한 데이터가 복원 후 더 이상 보이지 않아야 한다.
- A 시점에 이미 존재하던 데이터에 대해 B 시점에서 수정한 값이 복원 후 원래 값으로 돌아와야 한다.

즉, 단순 신규 생성 데이터 제거만 확인하지 않고, **A 데이터에 대한 수정 사항이 되돌아가는지**도 반드시 확인한다.

전체 흐름은 아래 순서를 따른다.

1. A 작업 진행
2. A 시점 수동 백업 수행
3. B 작업 진행
4. A 시점 백업본으로 복원 수행
5. 복원 후 시스템 상태가 B 작업 이전의 A 시점으로 돌아갔는지 확인

## 2. 사전 조건

- 관리자 계정으로 `백업 관리` 화면에 접근 가능해야 한다.
- 일반 사용자 계정 또는 관리자 계정으로 대상자 등록과 검사 수행이 가능해야 한다.
- 백업 ZIP 파일을 확보한 뒤 다시 업로드할 수 있어야 한다.
- 테스트 전 기존 데이터와 혼동되지 않도록 아래 식별자 규칙을 반드시 사용해야 한다.
- 백업/복원은 관리자 권한으로만 수행한다.
- 복원은 표준 전체 백업 ZIP 기준으로 진행한다.
- 복원 직전 자동 백업이 생성되는 구현을 전제로 검증한다.
- 복원 실행 시 현재 구현에서 확인 문구 입력이 요구되면, 현재 화면에 표시되는 실제 문구 기준으로 검증한다.
- 삭제 검증이 필요하더라도 하드 삭제 전제로 테스트하지 않는다. 신규 데이터 생성과 기존 데이터 수정 중심으로 검증한다.
- 서버가 ZIP 검증, 복원 가능 여부 판단, 자동 백업 생성, 실제 복원 실행을 담당한다는 전제로 진행한다.
- 가능하면 별도 브라우저 세션 2개를 준비한다.
- 브라우저 A: 관리자 로그인 유지용
- 브라우저 B: 회원가입 신청용(`로그아웃` 없이 신규 사용자 신청 처리)
- 공유 환경이면 테스트 중 다른 사용자의 동시 입력 작업이 없도록 시간대를 맞춘다.
- 이 문서에서는 **메뉴명/화면명 기준으로 먼저 이동하고, URL은 현재 구현 기준 예시 경로로만 참고**한다.

## 3. 테스트 데이터 네이밍 규칙

아래 값은 그대로 사용하는 것을 권장한다. A/B 식별자, 연락처 뒤 4자리, 메모 문구가 모두 다르므로 복원 전후 차이를 사람이 눈으로 바로 확인할 수 있다.

| 구분 | 필드 | 입력값 |
| --- | --- | --- |
| 대상자 A | 이름 | `RESTORE-A-CLIENT-001` |
| 대상자 A | 생년월일 | `1980.01.01` |
| 대상자 A | 연락처 | `010-7000-1001` |
| 대상자 A | B단계 수정 이름 | `RESTORE-A-CLIENT-001-BEDIT` |
| 대상자 A | B단계 수정 연락처 | `010-7000-9001` |
| 사용자 A | 이름 | `RESTORE-A-USER` |
| 사용자 A | 로그인 아이디 | `restore_a_user` |
| 사용자 A | 비밀번호 예시 | `RestoreA!123` |
| 사용자 A | 연락처 | `010-7100-1001` |
| 사용자 A | 소속 팀 | `복원테스트A팀` |
| 사용자 A | 가입 신청 메모 | `RESTORE-A-USER-MEMO` |
| A 검사 세션 | 척도 | `PHQ-9` |
| A 검사 세션 | 세션 메모 | `RESTORE-A-SESSION-MEMO-KEEP` |
| 대상자 B | 이름 | `RESTORE-B-CLIENT-001` |
| 대상자 B | 생년월일 | `1981.02.02` |
| 대상자 B | 연락처 | `010-7000-2001` |
| 사용자 B | 이름 | `RESTORE-B-USER` |
| 사용자 B | 로그인 아이디 | `restore_b_user` |
| 사용자 B | 비밀번호 예시 | `RestoreB!123` |
| 사용자 B | 연락처 | `010-7200-2001` |
| 사용자 B | 소속 팀 | `복원테스트B팀` |
| 사용자 B | 가입 신청 메모 | `RESTORE-B-USER-MEMO` |
| B 검사 세션 | 척도 | `GAD-7` |
| B 검사 세션 | 세션 메모 | `RESTORE-B-SESSION-MEMO-REMOVE` |

식별 원칙은 아래처럼 해석하면 된다.

- A 원본 데이터: 이름에 `RESTORE-A`, 연락처 뒤 4자리에 `1001`, 메모에 `KEEP`
- B 신규 데이터: 이름에 `RESTORE-B`, 연락처 뒤 4자리에 `2001`, 메모에 `REMOVE`
- B단계에서 A 데이터를 수정한 값: 이름 suffix `-BEDIT`, 연락처 뒤 4자리 `9001`
- 테스트 전 정리 단계에서 suffix를 추가했다면 A/B 구분 원칙만 유지한 채 같은 suffix 규칙을 함께 적용한다.

## 5. A 작업 단계

### 5.1 A-1 대상자 A 신규 등록

- [ ] 브라우저 A에서 대상자 등록 화면으로 이동한다.
- 화면 이동: 상단 메뉴 `대상자` -> `대상자 등록`
- 현재 구현 기준 예시 경로: `/clients/new`

입력 예시값

| 필드 | 값 |
| --- | --- |
| 이름 | `RESTORE-A-CLIENT-001` |
| 성별 | `남성` |
| 생년월일 | `1980.01.01` |
| 연락처 | `010-7000-1001` |
| 담당자 | 현재 로그인 사용자 기본값 사용 |

저장 후 기대 결과

- [ ] 저장 후 대상자 상세 화면으로 이동한다.
- [ ] 사례번호(`clientNo`)가 자동 생성된다.
- [ ] 상세 화면에 이름, 생년월일, 연락처, 상태가 표시된다.
- [ ] 상태는 활성 상태를 의미하는 현재 구현의 표현으로 보인다.

확인 방법

- 확인 화면 1: 대상자 상세 화면
- 현재 구현 기준 예시 경로: `/clients/:clientId`
- 확인 방법 1: 상세 화면에서 이름 `RESTORE-A-CLIENT-001`, 연락처 `010-7000-1001`, 상태, 사례번호를 확인한다.
- 확인 화면 2: 대상자 목록 화면
- 현재 구현 기준 예시 경로: `/clients`
- 확인 방법 2: 목록 화면으로 돌아가 이름 검색 `RESTORE-A-CLIENT-001` 후 검색 결과가 1건 이상 보이는지 확인한다.
- [ ] 운영자 기록표에 대상자 A 사례번호를 기록한다.

### 5.2 A-2 사용자 A 신규 등록

이 단계는 `회원가입 신청 -> 관리자 승인 -> 사용자 관리 화면 확인` 순서로 진행한다.

- [ ] 브라우저 B에서 회원가입 신청 화면을 연다.
- 화면 이동 1: 로그인 화면의 `회원가입 신청`
- 현재 구현 기준 예시 경로 1: `/signup`
- 화면 이동 2: 관리자 메뉴의 `회원가입 승인`
- 현재 구현 기준 예시 경로 2: `/admin/signup-requests`
- 화면 이동 3: 관리자 메뉴의 `사용자 관리`
- 현재 구현 기준 예시 경로 3: `/admin/users`

회원가입 신청 입력 예시값

| 필드 | 값 |
| --- | --- |
| 이름 | `RESTORE-A-USER` |
| 아이디 | `restore_a_user` |
| 비밀번호 | `RestoreA!123` |
| 비밀번호 확인 | `RestoreA!123` |
| 연락처 | `010-7100-1001` |
| 직책 또는 역할 | `실무자` |
| 소속 팀 | `복원테스트A팀` |
| 가입 신청 메모 | `RESTORE-A-USER-MEMO` |

저장 후 기대 결과

- [ ] 가입 신청 후 로그인 화면으로 이동하거나 신청 완료 안내 화면이 표시된다.
- [ ] 신청 접수 및 관리자 승인 필요를 의미하는 현재 구현의 안내 문구가 보인다.
- [ ] 관리자의 `회원가입 승인` 화면에 `restore_a_user` 신청 건이 보인다.
- [ ] 승인 후 `사용자 관리` 화면에서 `restore_a_user` 의 상태가 승인 완료/활성 상태를 의미하는 현재 구현 값으로 보인다.

확인 방법

- 확인 화면 1: 회원가입 승인 화면
- 현재 구현 기준 예시 경로: `/admin/signup-requests`
- 확인 방법 1: 신청 목록에서 이름 `RESTORE-A-USER`, 아이디 `restore_a_user`, 연락처 `010-7100-1001`, 신청 메모를 확인한 뒤 `승인` 처리한다.
- 확인 화면 2: 사용자 관리 화면
- 현재 구현 기준 예시 경로: `/admin/users`
- 확인 방법 2: 검색어에 `restore_a_user` 를 입력하고 조회한다.
- 기대 결과 2: 이름 `RESTORE-A-USER`, 아이디 `restore_a_user`, 연락처 `010-7100-1001`, 상태 `활성` 또는 동등한 현재 구현 표현이 표시된다.

### 5.3 A-3 대상자 A 검사 세션 1건 생성

- [ ] 대상자 A 상세 화면에서 검사 세션을 1건 저장한다.
- 화면 이동: 대상자 상세의 `검사 시작` -> `척도 선택` -> `입력` -> `요약` -> 저장 후 세션 상세
- 현재 구현 기준 예시 경로: `/clients/:clientId` -> `/assessments/start/:clientId/scales` -> `/assessments/start/:clientId/input` -> `/assessments/start/:clientId/summary` -> `/assessments/sessions/:sessionId`

입력 예시값

| 단계 | 값 |
| --- | --- |
| 척도 선택 | `PHQ-9` 1개 선택 |
| 문항 응답 | 1번~9번 모두 `전혀 아니다` 선택 |
| 세션 메모 | `RESTORE-A-SESSION-MEMO-KEEP` |

저장 후 기대 결과

- [ ] 세션 저장 후 세션 상세 화면으로 이동한다.
- [ ] 세션번호(`sessionNo`)가 생성된다.
- [ ] 세션 상세에 대상자명 `RESTORE-A-CLIENT-001` 이 표시된다.
- [ ] 세션 상세에 세션 메모 `RESTORE-A-SESSION-MEMO-KEEP` 이 표시된다.
- [ ] 세션 상세에 척도 `PHQ-9` 결과가 보인다.

확인 방법

- 확인 화면 1: 세션 상세 화면
- 현재 구현 기준 예시 경로: `/assessments/sessions/:sessionId`
- 확인 방법 1: 세션번호, 대상자명, 세션 메모, `PHQ-9` 결과를 확인한다.
- 확인 화면 2: 대상자 상세 화면
- 현재 구현 기준 예시 경로: `/clients/:clientId`
- 확인 방법 2: 대상자 상세의 `최근 검사 세션` 테이블에 방금 저장한 세션이 보이는지 확인한다.
- 확인 화면 3: 검사기록 목록 화면
- 현재 구현 기준 예시 경로: `/assessment-records`
- 확인 방법 3: 대상자명 `RESTORE-A-CLIENT-001` 로 검색했을 때 검사기록이 조회되는지 확인한다.
- [ ] 운영자 기록표에 A 세션 번호를 기록한다.

## 6. A 시점 수동 백업 단계

- [ ] 관리자 계정으로 백업 관리 화면에 진입한다.
- 화면 이동: 관리자 메뉴 `백업 관리`
- 현재 구현 기준 예시 경로: `/admin/backups`

실행 절차

1. `수동 백업 실행` 버튼 또는 동등한 현재 구현의 수동 백업 실행 버튼을 누른다.
2. `실행 사유` 또는 동등한 입력란에 `RESTORE-A-BASELINE-BACKUP` 을 입력한다.
3. 확인 모달에서 백업 실행을 확정한다.
4. 성공 안내, 최근 이력, 파일명, 파일경로를 확인한다.
5. 이후 복원 단계에서 사용할 수 있도록 **바로 이 A 시점 백업본** 을 확보한다.

저장 후 기대 결과

- [ ] 백업 실행 완료를 의미하는 현재 구현의 안내 문구가 표시된다.
- [ ] 안내 문구 또는 최신 이력에서 생성 파일명이 보인다.
- [ ] 백업 목록의 최신 행에 방금 실행한 이력이 보인다.
- [ ] 파일명은 표준 전체 백업 ZIP 형식의 `.zip` 파일이다.
- [ ] 이후 복원에 사용할 A 시점 ZIP 파일을 실제로 확보할 수 있다.

확인 방법

- 확인 화면 1: 백업 관리 화면 상단 안내 영역
- 현재 구현 기준 예시 경로: `/admin/backups`

반드시 기록할 항목

- [ ] A 시점 백업 파일명 기록
- [ ] 백업 목록 표시 여부 기록
- [ ] 다운로드 여부 기록
- [ ] 확보한 ZIP 경로 기록
- [ ] 이후 복원에서 **같은 파일명인지 재확인**

중요 메모

- 복원 단계에서 다른 최신 ZIP 을 사용하면 안 된다.
- 이 시나리오는 반드시 **A 시점에 생성한 동일한 ZIP 파일** 로 복원해야 한다.

## 9. 복원 실행 단계

- [ ] 관리자 계정으로 백업 관리 화면에 진입한다.
- 화면 이동: 관리자 메뉴 `백업 관리`
- 현재 구현 기준 예시 경로: `/admin/backups`
- [ ] **A 시점 백업 단계에서 확보한 동일 ZIP 파일** 을 사용한다.

중요 원칙

- 아래 상태값, 필드명, 확인 문구는 현재 구현 기준 예시이며, 실제 검증은 **현재 화면/응답에 표시되는 실제 상태값, 필드명, 문구 기준으로 판단**한다.
- 검증 완료를 의미하는 현재 화면/응답의 상태값 예시: `VALIDATED`
- 실제 복원 대상으로 표시되는 현재 화면의 항목명 예시: `DATABASE`
- 복원 성공을 의미하는 현재 화면/응답의 상태값 예시: `SUCCESS`
- 복원 검증 이력 식별 정보의 현재 화면/응답 필드명 예시: `restoreId`
- 복원 가능 항목 목록의 현재 화면/응답 필드명 예시: `detectedItems`
- 복원 직전 자동 백업 확인 정보의 현재 화면/응답 필드명 예시: `preBackupFileName`
- 현재 화면이 요구할 수 있는 확인 문구 예시: `전체 복원을 실행합니다`

실행 절차

1. `복원 ZIP 업로드` 영역에서 A 시점 ZIP 파일을 선택한다.
2. `업로드` 버튼을 눌러 ZIP 업로드와 서버 검증을 실행한다.
3. 성공 안내가 나오면 `복원 검증 이력` 목록에서 방금 업로드한 파일 행을 찾는다.
4. 해당 행을 클릭해 상세 화면 또는 상세 영역을 연다.
5. 파일명, 상태, 복원 검증 이력 식별 정보, 복원 가능 항목 표시 영역 또는 동등한 현재 구현의 상세 정보, 실행 가능 여부를 확인한다.
6. `복원 실행 준비` 영역에서 현재 화면에 표시되는 실제 복원 대상 항목을 선택한다.
- 예시: `DATABASE`
7. 확인 문구 입력란이 있다면 **현재 화면에 표시되는 확인 문구를 그대로** 입력한다.
- 예시: `전체 복원을 실행합니다`
8. 실행 전 요약에서 실행 준비 완료를 의미하는 현재 화면/응답의 상태가 보이는지 확인한다.
9. `복원 실행` 버튼을 누른다.
10. 실행 결과와 자동 백업 흔적을 확인한다.

복원 실행 중 필수 검증 포인트

- [ ] ZIP 업로드 가능 여부
- [ ] 복원 가능 항목 표시 여부
- [ ] 확인 문구 입력 요구 여부
- [ ] 복원 직전 자동 백업 생성 여부
- [ ] 복원 성공 메시지 또는 결과 화면 확인

상세 확인 방법

- 확인 화면 1: 백업 관리 화면의 `복원 ZIP 업로드`
- 현재 구현 기준 예시 경로: `/admin/backups`
- 확인 방법 1: A 시점 ZIP 파일이 선택되고 업로드가 성공하는지 확인한다.
- 기대 결과 1: 업로드/검증 완료를 의미하는 현재 구현의 성공 안내가 표시된다.

- 확인 화면 2: 백업 관리 화면의 `복원 검증 이력`
- 현재 구현 기준 예시 경로: `/admin/backups`
- 확인 방법 2: 방금 업로드한 파일명과 동일한 행을 찾는다.
- 기대 결과 2: 상태가 **검증 완료를 의미하는 현재 화면/응답의 실제 값** 이어야 한다.
- 예시: `VALIDATED`
- 기대 결과 2-1: 실행 가능 여부가 현재 구현 기준으로 복원 가능 상태여야 한다.

- 확인 화면 3: 백업 관리 화면의 복원 상세 영역
- 현재 구현 기준 예시 경로: `/admin/backups`
- 확인 방법 3: 방금 선택한 행의 파일명이 A 시점 백업 파일명과 정확히 일치하는지 확인한다.
- 기대 결과 3: 복원 검증 이력 식별 정보가 보이거나 동등한 현재 구현의 상세 정보에서 확인 가능해야 하고, 복원 가능 항목 또는 상세 정보에서 실제 복원 대상 항목이 확인 가능해야 한다.
- 예시 식별 정보: `restoreId`
- 예시 항목/필드: `DATABASE`, `db/database.sql`, `detectedItems`

- 확인 화면 4: 백업 관리 화면의 `복원 실행 준비`
- 현재 구현 기준 예시 경로: `/admin/backups`
- 확인 방법 4: 현재 구현에서 실제 선택 가능한 복원 항목을 선택하고, 현재 화면이 요구하는 확인 문구를 입력한다.
- 기대 결과 4: 확인 문구를 정확히 입력하기 전에는 실행 준비가 완료되지 않고, 정확히 입력하면 실행 준비 완료를 의미하는 현재 화면/응답의 상태로 바뀐다.
- 예시 확인 문구: `전체 복원을 실행합니다`

- 확인 화면 5: 백업 관리 화면의 실행 후 상세/요약
- 현재 구현 기준 예시 경로: `/admin/backups`
- 확인 방법 5: 복원 완료 후 성공 메시지와 복원 직전 자동 백업이 생성되었음을 보여주는 현재 구현의 화면 요소 또는 상세 정보를 확인한다.
- 기대 결과 5: 복원 완료를 의미하는 현재 화면/응답의 안내가 보여야 하며, 자동 백업이 생성되었음을 확인할 수 있어야 한다.
- 예시 확인 정보: `preBackupFileName`

중단 기준

- [ ] 업로드 후 상태가 검증 완료를 의미하는 현재 화면/응답의 실제 값이 아니면 중단한다.
- 예시: `VALIDATED`
- [ ] 실행 가능 여부가 실행 불가 상태면 중단하고 사유를 기록한다.
- [ ] 현재 화면/응답에서 실제 복원 대상으로 선택해야 하는 항목이 보이지 않으면 중단한다.
- 예시: `DATABASE`
- [ ] A 시점 백업 파일명이 아니면 중단한다.

## 10. 복원 직후 화면 정리

복원 직후에는 화면 캐시, 목록 필터, 인증 상태 때문에 실제 복원 상태와 다르게 보일 수 있으므로 아래 순서를 먼저 수행한다.

- [ ] 대상자 목록 화면 전체 새로고침
- [ ] 사용자 관리 화면 전체 새로고침
- [ ] 검사기록 목록 화면 전체 새로고침
- [ ] 각 목록 화면의 검색어, 날짜, 상태 필터, 체크박스를 초기화한 뒤 다시 조회
- [ ] 복원 상세나 결과 화면이 남아 있으면 백업 관리 화면도 한 번 재조회
- [ ] 화면이 비정상적으로 남아 있거나 권한 상태가 애매하면 로그아웃 후 관리자 재로그인

권장 확인 화면

- 대상자 목록 화면
- 사용자 관리 화면
- 검사기록 목록 화면
- 백업 관리 화면

## 11. 복원 후 검증 항목

아래 표를 모두 확인한다.

| 검증 항목 | 확인 화면 | 확인 방법 | 기대 결과 |
| --- | --- | --- | --- |
| A 대상자가 여전히 존재하는가 | 대상자 목록, 대상자 상세 | 이름 `RESTORE-A-CLIENT-001` 로 검색하고 상세를 연다. | 대상자 A가 존재한다. |
| A 사용자 계정이 여전히 존재하는가 | 사용자 관리 화면 | 검색어 `restore_a_user` 로 조회한다. | 사용자 A가 조회되고 상태가 승인 완료/활성 상태를 의미하는 현재 구현 표현이다. |
| A 검사 세션이 A 시점 기준으로 존재하는가 | 대상자 상세, 세션 상세, 검사기록 목록 | 대상자 A 상세의 최근 세션 목록 또는 검사기록 목록에서 A 세션을 열고 세션 메모를 확인한다. | `RESTORE-A-SESSION-MEMO-KEEP` 세션이 존재한다. |
| B 대상자가 사라졌는가 | 대상자 목록 | 이름 `RESTORE-B-CLIENT-001` 로 검색한다. | 검색 결과가 0건이다. |
| B 사용자 계정이 사라졌는가 | 사용자 관리 화면 | 검색어 `restore_b_user` 로 조회한다. | 검색 결과가 0건이거나, 현재 구현상 조회되더라도 A 시점에 존재하지 않았던 신규 B 사용자로 남아 있지 않다. |
| B 검사 세션이 사라졌는가 | 검사기록 목록 | 대상자명 `RESTORE-B-CLIENT-001` 로 조회한다. | 검색 결과가 0건이다. |
| B 단계에서 수정했던 A 대상자 정보가 원래 값으로 복구되었는가 | 대상자 상세 | A 대상자 상세에서 이름과 연락처를 확인한다. | 이름은 `RESTORE-A-CLIENT-001`, 연락처는 `010-7000-1001` 로 돌아온다. `RESTORE-A-CLIENT-001-BEDIT`, `010-7000-9001` 은 보이지 않아야 한다. |
| 백업/복원 이력 또는 자동 백업 생성 흔적이 관리자 화면에서 확인되는가 | 백업 관리 화면 | 복원 상세의 자동 백업 관련 정보와 복원 성공 상태를 확인하고 필요 시 백업 이력도 재조회한다. | 복원 직전 자동 백업이 생성되었음을 보여주는 현재 구현의 화면 요소 또는 상세 정보가 보이고, 복원 이력이 성공 상태로 확인된다. |

추가 확인 권장 항목

- [ ] A 대상자 상세의 사례번호가 A 작업 때 기록한 값과 같은지 확인
- [ ] A 세션 번호가 A 작업 때 기록한 값과 같은지 확인
- [ ] B 데이터가 단순히 필터에서 숨겨진 것이 아닌지, 검색 조건 초기화 후에도 조회되지 않는지 확인

선택 검증

아래 항목은 선택 검증이다. 수행이 어렵더라도 최종 PASS/FAIL 판정은 위 표의 핵심 복원 항목 기준으로 판단한다.

- [ ] 선택 검증 1: 가능하면 로그아웃 후 `restore_b_user` 로 로그인 시도한다.
- 기대 결과 1: 복원 후 B 사용자가 남아 있지 않아 로그인 실패해야 한다.
- [ ] 선택 검증 2: 가능하면 로그아웃 후 `restore_a_user` 로 로그인 시도한다.
- 기대 결과 2: 현재 정책상 승인된 활성 사용자로 남아 있다면 로그인 가능해야 한다.
- [ ] 선택 검증 3: 로그인 검증이 어렵다면 사용자 관리 화면에서 A 사용자 상태가 승인 완료/활성 상태인지 다시 확인한다.

## 12. 증적 수집 항목

증적은 너무 과하지 않게 모으되, 실패 원인 추적이 가능한 수준으로 남긴다. 아래 최소 증적을 권장한다.

- [ ] 테스트 시작 전 기준 화면
- 대상자 목록 또는 백업 관리 화면에서 현재 테스트 식별자가 없음을 보여주는 화면

- [ ] A 작업 완료 화면
- 대상자 A 상세 화면
- 사용자 A가 승인 완료/활성 상태로 보이는 사용자 관리 화면
- A 세션 상세 화면

- [ ] A 백업 생성 목록
- 백업 관리 화면의 성공 안내와 최신 백업 이력 행
- 파일명과 시작/완료 시각이 함께 보이도록 캡처

- [ ] B 작업 완료 화면
- 대상자 B 상세 화면
- 사용자 B가 승인 완료/활성 상태로 보이는 사용자 관리 화면
- A 대상자 수정 후 상세 화면
- B 세션 상세 화면

- [ ] 복원 직전 자동 백업 생성 화면
- 복원 실행 후 백업 관리 상세의 자동 백업 관련 정보
- 예시 확인 정보: `preBackupFileName`
- 가능하면 복원 직후 백업 이력 재조회 화면

- [ ] 복원 성공 결과 화면
- 복원 성공을 의미하는 현재 구현의 메시지
- 복원 검증 상세의 성공 상태 또는 동등한 현재 구현 표현
- 예시 상태값: `SUCCESS`

- [ ] 복원 후 A/B 데이터 비교 화면
- A 대상자 상세에서 원래 이름/연락처로 돌아온 화면
- 대상자 목록에서 B 대상자 검색 결과 0건 화면
- 사용자 관리 화면에서 B 사용자 검색 결과 0건 또는 미존재 판단 근거 화면
- 검사기록 목록에서 B 대상자 검사기록 검색 결과 0건 화면

캡처 시 주의사항

- 가능하면 화면 상단 URL 또는 메뉴명, 검색어, 결과 영역이 함께 보이게 캡처한다.
- 같은 화면을 여러 번 캡처할 때는 복원 전/복원 후가 구분되도록 파일명 또는 캡처 순서를 관리한다.

## 13. 실패 시 점검 포인트

- 관리자 권한으로 접속했는지 확인
- 선택한 ZIP 파일이 **정말 A 시점 백업본** 인지 파일명으로 재확인
- 복원 직전 자동 백업 생성이 실패했는지 현재 구현의 자동 백업 관련 상세 정보로 확인
- 예시 확인 정보: `preBackupFileName`
- 목록이 갱신되지 않은 단순 캐시/새로고침 문제인지 확인
- 검색 조건, 날짜 필터, 상태 필터가 남아 있지 않은지 확인
- 업로드한 ZIP 검증 상태가 검증 완료를 의미하는 현재 화면/응답의 실제 값인지 확인
- 예시 상태값: `VALIDATED`
- 현재 화면/응답에서 실제 복원 대상으로 선택해야 하는 항목이 선택되었는지 확인
- 예시 항목: `DATABASE`
- 확인 문구가 현재 화면에서 요구하는 문구와 정확히 일치했는지 확인
- 예시 문구: `전체 복원을 실행합니다`

## 14. 최종 판정 기준

현재 구현의 상태값, 필드명, 버튼명, 안내 문구 표현이 예시와 조금 다르더라도, **실질적으로 B 작업이 사라지고 A 시점 상태가 복구되었는지**를 중심으로 판정한다.

### PASS 조건

아래 조건을 모두 만족하면 `PASS` 로 판정한다.

- [ ] 현재 구현의 표현 차이와 무관하게 B 신규 데이터가 실질적으로 남아있지 않다.
- [ ] 현재 구현의 표현 차이와 무관하게 A 시점 데이터가 보존된다.
- [ ] B 단계에서 수정한 A 데이터가 원래 값으로 돌아온다.
- [ ] 복원 전 자동 백업 생성 흔적이 현재 구현의 화면 요소 또는 상세 정보로 확인된다.
- [ ] 복원 실행 결과가 현재 구현의 성공 표현으로 확인된다.

### FAIL 조건

아래 중 하나라도 발생하면 `FAIL` 로 판정한다.

- [ ] B 대상자, B 사용자, B 세션 중 하나라도 복원 후 계속 남아 있다.
- [ ] A 대상자, A 사용자, A 세션 중 하나라도 복원 후 사라지거나 일부 값이 달라진다.
- [ ] A 대상자 수정값이 원래 값으로 돌아오지 않는다.
- [ ] 복원 직전 자동 백업 흔적이 없다.
- [ ] A 시점 백업본이 아닌 다른 ZIP 으로 복원했다.
- [ ] 현재 구현의 표현 차이를 감안해도 실질 상태가 A 시점으로 복구되었다고 보기 어렵다.
